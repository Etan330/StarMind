Product Requirements Document / V0.5

StarMind：主动知识筛选与边界链接同步 Agent

StarMind 面向高频收藏但无力整理的知识工作者。收藏夹已经代表用户做过一次粗筛，因此 StarMind 不做复杂价值评分， 只判断内容是否属于知识/干货类。知识类内容进入“原始资料库”，原始资料永久保留、不可被 Agent 删除。 增量同步不再依赖时间戳作为主逻辑，而是记录上一次扫描时收藏列表顶部或最后处理到的“边界链接”。下一次扫描时，从最新收藏开始读取，直到遇到已存在于原始资料库或同步账本中的边界链接即停止。

0. 本版核心修订

修订项 上一版问题 V0.3 修订方案
评分机制 使用多维 KWS 评分，过度复杂，且忽略“用户收藏已经是一次粗筛”。 改为单维度判断：是否为知识/干货类内容。知识类直接入库；文娱类剔除。
知识维护 只写了摘要、标签、图谱，缺少可解释的自主迭代逻辑。 引入 Karpathy LLM Wiki 模式：Raw Sources → LLM Wiki → Ingest / Query / Lint 循环。
同步去重 没有明确说明定期扫描时如何避免重复读取。 新增 Sync Ledger。用平台收藏时间戳/扫描时间戳做增量扫描，不依赖知识库内容做去重。
知识库删除问题 如果用知识库做去重，知识自主迭代删除后会导致旧内容被重复抓取。 Wiki/SOP 与 Raw Sources/Sync Ledger 分离。即使派生知识被删除，Raw Sources 和 Sync Ledger 仍能作为稳定停止依据。

0.1 本版新增核心原则：原始资料不可变，派生知识可迭代

StarMind 必须把“原始资料”和“沉淀后的方法论/SOP”彻底分层。 用户收藏进来的原始文章、视频逐字稿、图片 OCR、PDF、帖子、网页快照等内容，属于用户的原始资料资产，不应被 Agent 删除、覆盖或重写。 Agent 的删除、合并、重写、覆盖、迭代，只能发生在派生知识层，例如主题页、概念页、方法论页、SOP 页、总结页和索引页。

为什么必须分层

层级 用户预期 Agent 权限 原因
原始资料层 “我收藏过的原始内容应该还在。” 只能追加、索引、引用，不能删除或覆盖。 原始资料是事实来源和用户资产，用户可能未来还要查看原文、逐字稿、截图或文件。
派生知识层 “帮我把资料沉淀成更好的知识结构。” 可以合并、重写、删除、覆盖、重构。 方法论、SOP、主题总结本来就是可迭代产物，需要持续优化。

新的维护边界

Agent 可以做：
- 合并两个重复的主题页
- 重写一个方法论页
- 更新一个 SOP 步骤
- 删除过时的总结段落
- 重构索引和知识图谱关系
- 把多个来源提炼成新的最佳实践

Agent 不可以做：
- 删除用户收藏过的原始链接
- 删除原始文章正文
- 删除视频逐字稿
- 覆盖原始 OCR/PDF 文本
- 修改 Raw Source 的事实内容
- 因为 Wiki 页面被合并而删除原始资料

1. 产品定位

StarMind 是一个主动知识筛选与自维护知识库 Agent。 它持续同步用户授权平台中的收藏内容，只过滤掉非知识类、文娱类、低信息量内容； 对知识/干货类内容自动转化为不可变 Raw Sources，并由 LLM 编译成持续更新的 Wiki 知识层。

一句话定义

StarMind 把用户分散在各平台收藏夹里的知识类内容，自动筛选、编译并维护成一个持续迭代的个人 LLM Wiki。

产品愿景

让收藏夹从“沉睡的信息仓库”变成“会自我编译、自我维护、自我进化的知识系统”。 用户只负责收藏和提出问题；StarMind 负责筛选知识、维护结构、更新连接、发现缺口。

核心用户

用户 典型收藏内容 核心痛点 StarMind 价值
学习/研究型用户 课程视频、论文解读、公众号文章、知乎回答、B站教程。 收藏很多，但无法形成体系。 筛出知识类内容，自动形成主题 Wiki。
产品/创业/商业分析用户 行业分析、竞品案例、融资新闻、专家观点。 信息分散，难以沉淀为长期洞察。 自动编译为行业主题页、案例页和对比页。
开发者/AI 工具用户 GitHub Stars、技术文章、YouTube/B站教程、工具测评。 知识源分散，无法持续维护学习路线。 把项目、教程、概念、经验链接成知识图谱。
创作者/内容生产者 选题、案例、观点、素材、数据。 素材沉睡，产出时找不到。 自动变成选题库、观点库、案例库。

2. 核心问题定义

01
收藏夹不是知识库

收藏夹混有知识、娱乐、消费和临时兴趣，不能全量入库。

02
用户已经做过粗筛

收藏行为本身已经说明用户认为内容可能有用，系统不应再复杂打分。

03
维护才是长期成本

真正让知识库失效的是后续没人更新、没人去重、没人维护连接。

04
同步不能依赖知识库

知识页可能被删除、合并、重写，因此不能用知识库内容判断是否已抓取。

关键产品判断： 收藏夹中的知识类/干货类内容应直接进入知识库；评分机制只承担“剔除文娱类、消费类、低信息量内容”的职责。

3. 知识分类器：只判断是不是知识

V0.3 不再使用复杂多维评分。系统只输出一个核心字段：is_knowledge。 可以保留 confidence 作为工程阈值，但它不是“价值评分”，只表示模型对“是否属于知识类内容”的判断置信度。

3.1 分类标签

标签 含义 处理策略
knowledge 知识类、干货类、教程类、方法论类、行业分析类、案例类、资料类内容。 直接进入 Raw Sources，并触发 LLM Wiki Ingest。
uncertain 可能有知识价值，但标题/元数据不足，或内容偏经验/观点/测评。 进入待确认区；也可先轻量归档，不进入核心 Wiki。
non_knowledge 文娱、搞笑、明星、纯消费、抽奖、低信息量、纯情绪表达内容。 剔除或归档，不进入知识库。

3.2 什么算知识类/干货类

知识类型 判断标准 示例
概念解释 解释某个概念、框架、原理、机制。 “什么是 RAG”“Agentic Workflow 详解”。
教程/操作方法 教用户完成某个任务，包含步骤、流程、配置、工具。 “用 n8n 自动同步飞书多维表格”。
经验总结 对某类实践进行复盘、避坑、经验归纳。 “小团队做 AI 产品的 10 个坑”。
行业/商业分析 包含趋势判断、案例、数据、商业模式、竞争格局。 “2026 年具身智能融资格局”。
研究资料 论文、报告、文档、长访谈、公开课、技术博客。 “Karpathy LLM Wiki 思路解析”。
可复用素材 对创作、研究、写作、决策有复用价值。 案例库、数据图表、产品截图、竞品拆解。

3.3 什么应被剔除

类型 判断标准 默认处理
纯娱乐 搞笑、舞蹈、段子、明星动态、综艺切片。 non_knowledge，剔除。
纯消费 商品种草、折扣、购物链接、无分析的测评。 non_knowledge，归档或剔除。
低信息量 只有情绪表达、口号、标题党，无可复用信息。 non_knowledge，剔除。
短时效资讯 无长期价值的临时热点、无分析的新闻快讯。 默认剔除；若是行业分析则保留。
重复搬运 同一内容多个平台重复出现且无新增信息。 不进入 Raw Sources，记录到 Sync Ledger。

3.5 工程阈值

置信度 标签 动作
confidence ≥ 0.75 knowledge 自动进入 Raw Sources，触发 Ingest。
0.45 ≤ confidence < 0.75 uncertain 进入待确认区，或轻量归档。
confidence < 0.45 non_knowledge 剔除，不进入知识库。

4. Karpathy LLM Wiki 维护逻辑

StarMind 的知识维护不应停留在“摘要 + 标签 + 向量库”。V0.3 采用 Karpathy LLM Wiki 的核心思想： 生材料是不可变的 Raw Sources，LLM 将其编译成结构化 Wiki，并通过 Ingest、Query、Lint 三个操作持续迭代。

4.1 四层结构

层级 内容 Agent 是否可修改 StarMind 中的实现
Raw Sources / 原始资料层 文章正文、视频逐字稿、图片 OCR、PDF、帖子、网页快照、代码项目元信息。 不可修改、不可删除。只能追加、索引、引用。 原始资料文件夹。作为事实底座和用户资产长期保留。
Source Summary / 来源摘要层 每条原始资料对应的摘要、要点、关键词、引用位置。 可重新生成，但必须保留版本记录。 方便快速阅读原始资料，也为后续 Wiki 编译提供中间层。
Wiki / Methodology / SOP 派生知识层 主题页、概念页、方法论页、SOP、对比页、索引页、最佳实践。 可以合并、删除、覆盖、重写、重构。 Agent 主要维护对象。Karpathy 式自主迭代只发生在这一层。
Schema / 规则层 页面命名、标签体系、引用规范、合并规则、删除权限、SOP 格式。 由人定义，Agent 遵守；重大修改需用户确认。 StarMind Knowledge Schema / Agent Instructions。

4.2 三个核心操作

操作 含义 StarMind 中的具体动作
Ingest 将新的 Raw Source 编译进 Wiki。 生成 source summary，更新相关主题页/概念页，补充索引，写入操作日志。
Query 用户或 Agent 对 Wiki 提问。 基于 Wiki 回答问题；高价值回答可写回 Wiki，成为新页面或补充段落。
Lint 定期健康检查。 检查矛盾、过期信息、孤立页面、缺失链接、重复页面、知识缺口，并提出修复。

4.3 StarMind 的维护循环

新收藏内容
→ Knowledge Classifier 判断是否为知识
→ 知识类内容进入 Raw Sources
→ Ingest Agent 编译进 Wiki
→ 更新 Source Summary / Topic Page / Concept Page / Index / Log
→ 用户提问或系统主动分析
→ Query 结果中有价值的内容写回 Wiki
→ 定期 Lint 检查矛盾、过期、孤立、重复、缺口
→ Repair Agent 自动修复低风险问题，高风险问题进入 Review Queue
关键原则： Raw Sources 是事实底座和用户资产，不能被删除、覆盖或重写；Wiki / Methodology / SOP 是 LLM 编译层，可以被更新、合并、删除、重构。 因此同步去重不能依赖 Wiki 或知识页是否存在，也不能因为派生知识层被删除而删除原始资料。

5. 总体架构图

5.1 数据生命周期与删除权限

StarMind 中的“删除”必须分层处理。系统允许 Agent 删除或覆盖派生知识，但不允许 Agent 删除原始资料。 只有用户本人可以主动删除原始资料，且删除时需要明确提示这会影响后续可追溯性。

对象 保存位置 是否可被 Agent 删除 是否可被用户删除 说明
原始链接 / 原始文件 Raw Sources Folder 否 是 用户资产，长期保留。
原始正文 / 逐字稿 / OCR / PDF 文本 Raw Sources Folder 否 是 事实来源，不参与自动删除。
Source Summary Source Summary Folder 可重新生成，但不直接物理删除 是 作为原始资料的阅读索引，可保留版本。
主题页 / 概念页 Wiki Folder 是 是 可被合并、重写、删除。
方法论 / SOP Methodology / SOP Folder 是 是 Agent 自主迭代的核心对象。
索引 / 知识图谱关系 Index / Graph DB 是 是 可根据最新知识结构重构。
Sync Ledger Sync DB 否 一般不暴露给用户删除 用于增量扫描和去重，独立于知识库。

维护动作边界

允许 Agent 自动执行：
- 合并重复的主题页
- 删除过时的方法论段落
- 重写 SOP
- 将多个零散总结合并为一个最佳实践
- 重建索引和知识图谱边
- 标记某个 Source Summary 需要重新生成

不允许 Agent 自动执行：
- 删除 Raw Source
- 覆盖 Raw Source 原文
- 删除用户原始收藏链接
- 因为内容被判定过时而删除原始资料
- 因为 SOP 被重写而删除支持它的来源材料

版本化要求

派生知识层的每次重大改写都应该记录版本，包括改写原因、引用了哪些 Raw Sources、删除或合并了哪些旧段落。 这样可以让用户知道 SOP 是如何从原始资料中沉淀出来的。

  "target_id": "sop_agentic_knowledge_loop",
  "operation": "rewrite",
  "reason": "新增 3 条 Raw Sources 后，旧 SOP 的步骤顺序不再合理。",
  "source_refs": ["raw_001", "raw_009", "raw_018"],

6. 增量同步：从时间戳水位改为边界链接停止爬取

V0.5 将同步策略从“时间戳水位”调整为“边界链接水位”。原因是很多平台收藏页不一定稳定暴露收藏时间， 但收藏列表通常会按最近收藏排序。系统可以记录上一次扫描时的边界链接，下次从最新内容开始扫描， 一旦遇到已经存在于原始资料库或同步账本中的链接，就停止继续爬取。

新的主逻辑： 每轮扫描从收藏页顶部开始读取，逐条标准化 URL。如果发现当前 URL 已存在于 Raw Sources / Sync Ledger， 即认为已经扫描到旧内容边界，可以停止继续翻页。新出现的链接进入 Candidate Pool。

6.1 为什么可以用原始资料库做停止判断

之前不建议用知识库做去重，是因为 Wiki / SOP / 方法论层会被 Agent 删除、合并、重写。 但 V0.4 已经明确：Raw Sources 原始资料层不可被 Agent 删除。因此，Raw Sources 可以作为“已采集原始资料”的稳定判断依据。

对象 是否适合作为同步停止依据 原因
Wiki 页面 / SOP / 方法论 不适合 这些内容会被 Agent 删除、合并、覆盖，不能代表原始链接是否采集过。
Raw Sources 原始资料库 适合 原始资料不可被 Agent 删除，能稳定记录已采集过的原始链接。
Sync Ledger 同步账本 适合 记录所有已扫描过的 external_item_id / canonical_url，即使内容后来被判定为非知识也可避免重复扫描。

6.4 边界链接扫描逻辑

For each connector:
  1. 打开用户收藏页
  2. 从页面顶部开始读取收藏项
  3. 对每个 item 提取 external_item_id 和 canonical_url
  4. 检查 canonical_url / external_item_id 是否存在于：
     - Raw Sources
     - Sync Ledger scan_item_log
  5. 如果存在：
     - 认为已到达上次扫描边界
     - 停止继续翻页
  6. 如果不存在：
     - 写入 scan_item_log
     - 进入 Candidate Pool
  7. 本轮扫描结束后：
     - 将本轮第一个新链接记录为 last_top_url
     - 将命中的旧链接记录为 last_boundary_url
     - 如果本轮没有命中旧链接，则触发安全兜底策略

6.5 URL 标准化规则

使用链接作为边界时，必须做 URL 标准化，否则同一个内容可能因为参数不同被识别为不同链接。 

canonical_url 规则：
- 去掉追踪参数：utm_source、spm_id_from、share_source、timestamp 等
- 统一 http/https
- 统一移动端/桌面端域名
- 提取平台稳定 ID：
  - B站：BV ID
  - YouTube：video_id
  - 抖音：aweme_id / video_id
  - 小红书：note_id
  - 知乎：answer_id / article_id
- 优先使用 platform + external_item_id 作为唯一键
- canonical_url 作为辅助唯一键

6.6 安全兜底策略

异常情况 风险 处理策略
本轮一直没有遇到旧链接 可能是首次扫描、列表排序变化、平台页面结构变化，或用户大量新增收藏。 最多扫描 max_scan_pages；超限后停止，并提示“未命中历史边界”。
收藏列表排序不是按最近收藏 边界链接停止逻辑可能漏抓。 该平台降级为固定页数扫描 + scan_item_log 去重。
旧链接被用户从平台收藏夹删除 找不到上次边界。 继续扫描到命中任意已存在 Raw Source / Sync Ledger 项，或达到 max_scan_pages。
同一内容 URL 参数变化 重复采集。 使用 external_item_id 优先去重，canonical_url 辅助去重。
内容已扫描但被判定为 non_knowledge Raw Sources 中不存在，若只查 Raw Sources 会重复扫描。 必须同时查询 Sync Ledger。Sync Ledger 记录所有扫描过的链接，不管是否入库。
最终推荐： 停止爬取时不要只查 Raw Sources，也要查 Sync Ledger。 因为有些内容可能被判定为 non_knowledge，没有进入 Raw Sources，但仍然已经被扫描过。 最稳的判断是：canonical_url / external_item_id 存在于 Raw Sources 或 Sync Ledger 任一处，即可停止或跳过。

7. 关键流程

7.1 主动同步流程

用户勾选平台并授权
→ Scheduler 定时触发 Connector
→ Connector 从收藏页顶部开始读取
→ 遇到 Raw Sources / Sync Ledger 中已存在的边界链接即停止
→ 写入 scan_item_log
→ 内容进入 Candidate Pool
→ Knowledge Classifier 判断是否为知识
→ knowledge 进入 Raw Sources
→ Ingest Agent 编译到 LLM Wiki
→ non_knowledge 被剔除或归档

7.2 被动输入流程

用户发送公众号文章/链接/文件
→ Passive API 接收
→ 写入 scan_item_log，source_type = passive
→ Knowledge Classifier 判断是否为知识
→ 知识类内容进入 Raw Sources
→ Ingest Agent 更新 Wiki
→ 非知识内容忽略

7.3 LLM Wiki 自主维护流程

Daily / Weekly Maintenance:
  1. 读取 wiki/index.md 和最近 log
  2. 检查新旧页面之间的重复、冲突、孤立节点
  3. 合并重复主题页
  4. 给缺少链接的页面补 cross-reference
  5. 标记过期信息或被新来源覆盖的判断
  6. 生成“下一步应该研究的问题”
  7. 只对 Wiki / Methodology / SOP 层执行删除、合并、覆盖
  8. Raw Sources 永不被 Agent 删除
  9. 高风险修改进入 Review Queue
  10. 低风险修复自动写入派生知识层

9. Agent 模块

Agent 职责 关键输出
Connector Agent 按照 Sync Ledger 的时间水位扫描新增收藏。 CandidateItem、scan_item_log。
Knowledge Classifier 判断内容是否属于知识/干货类。 knowledge / uncertain / non_knowledge。
Extractor Agent 抽取正文、逐字稿、OCR、PDF 文本。 RawSource clean_text。
Ingest Agent 将 Raw Source 编译进 Wiki。 source summary、topic page updates、index、log。
Query Agent 基于 Wiki 回答用户问题，并把高价值回答写回。 answer page、comparison page、new concept page。
Lint Agent 周期性维护 Wiki 健康。 stale claims、orphan pages、missing links、knowledge gaps。

10. MVP 范围

10.1 必做能力

模块 功能 优先级
主动 Connector B站、抖音、小红书、YouTube 中至少 2-3 个。 P0

11. 路演话术

StarMind is not trying to save everything. It only turns knowledge-like saves into a self-maintaining LLM Wiki.

用户的收藏夹已经经过用户筛选，所以 StarMind 不需要复杂地判断“这个内容值不值得”。 我们只判断一个问题：它是不是知识/干货。是知识，就进入 Raw Sources 并由 LLM 编译进 Wiki； 不是知识，比如娱乐、消费、低信息量内容，就剔除。后续我们借鉴 Karpathy 的 LLM Wiki 思路， 让 LLM 持续维护 index、summary、concept pages 和 log，通过 Ingest、Query、Lint 形成自主迭代。 同时，我们用独立 Sync Ledger 做时间戳增量同步，不依赖知识库内容去重，避免知识页被删除或合并后重复抓取。
