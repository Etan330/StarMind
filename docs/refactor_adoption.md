# StarMind 重构方案采纳评估

这份评估只按用户视角判断：功能是否真的有用，表达是否清晰，是否能减少用户操作成本。

## 已采纳并落地

### 1. 状态词表集中管理

采纳原因：用户不应该被工程状态绕晕。候选内容、待确认、回收站、原始资料之间必须有清晰生命周期。

已落地：

- 新增 `app/services/statuses.py`
- 统一 `pending_classification`、`classified_knowledge`、`classified_uncertain`、`archived_recoverable`、`ingested`、`recycled` 等状态
- 待处理页不再只识别一个旧状态

### 2. 知识分类器

采纳原因：StarMind 的核心不是让用户逐条手动判断收藏，而是让 Agent 先筛掉大部分内容。用户只处理不确定项。

已落地：

- 新增 `ClassifierService`
- 复用已有 `classify_knowledge.md` prompt
- 写入已有 `KnowledgeClassification` 表
- 高置信知识进入原始资料
- 不确定内容留在待确认
- 明显非知识进入可恢复回收站
- LLM 失败时安全降级，不误删、不崩溃

### 3. 可恢复回收站

采纳原因：误判一定会发生。用户需要能把内容从回收站捞回来，而不是进入死胡同。

已落地：

- 新增 `RecycleService`
- 新增恢复接口
- 待处理页回收站支持：
  - 恢复到待确认
  - 恢复并标为知识

### 4. 待处理页表达优化

采纳原因：用户要一眼理解自己该做什么。

已落地：

- 增加“让 Agent 先筛一遍”
- 文案改为“先让 Agent 自动筛选，用户只处理不确定内容”
- 生成类按钮增加处理中状态，避免用户以为点击无效

### 5. 同步后的默认处理逻辑

采纳原因：同步收藏夹后，用户不应该马上面对一堆未处理链接。

已落地：

- 抖音同步后会触发分类路由
- 高置信内容进入原始资料
- 不确定内容留在待处理
- 非知识内容进入可恢复回收站

## 暂缓采纳

### 1. 完整七阶段大重构

暂缓原因：方向正确，但一次性重写路由、模板、服务层、Agent runner、设计系统，风险太大。当前项目已经有能跑的本地链路，应该先补核心断点。

后续可逐步推进：

- Lint / Repair
- Revision 回滚
- Source Summary 层
- Query 流式引用
- Wiki 自维护

### 2. 删除所有旧 CSS 并重建设计系统

暂缓原因：从产品角度需要做，但不是这次最阻塞用户使用的点。现在更重要的是让同步、分类、回收、沉淀的链路先正确。

### 3. 工作台、军团、激活的完整后端

暂缓原因：这些功能有想象力，但还不如分类器、回收站、知识沉淀重要。先保证“收藏不吃灰”的主链路。

### 4. 复杂 Lint / 自动修复

暂缓原因：这属于高风险能力。自动改知识库需要 Revision、审核、回滚一起上线，否则会让用户不放心。

## 当前用户可理解的主流程

```text
同步收藏夹
  -> Agent 自动筛选
  -> 高置信知识进入原始资料
  -> 不确定内容进入待确认
  -> 非知识内容进入可恢复回收站
  -> 用户从原始资料继续沉淀为知识库 / 方法论 / SOP
```

## 下一步建议

1. 接入真实 ASR，让原始资料不只是页面文本草稿。
2. 做 `Source Summary` 层，把原始资料和 Wiki 之间的关系讲清楚。
3. 做 Revision 回滚，再上线 Lint / Repair。
4. 把 Query 做成带引用、可写回 Wiki 的真正知识库对话。
